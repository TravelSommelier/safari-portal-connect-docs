Safari Portal Connect SDK

Internal reference for the TypeScript client we use to talk to the Safari Portal Connect API. Fully typed, handles auth and token refresh for you, and gives you arrays by default with lazy streaming when you want it.

Overview

The SDK wraps the Connect API (accounts, contacts, files, tasks, users) behind a typed client. You authenticate once, scope to an account, then call resources. It deals with OAuth token refresh, pagination, retries, and timeouts so you do not have to.

  • Typed end to end, including the responses and the query filters.
  • Array-first. list() returns every item across all pages.
  • Resilient. Per-request timeout, retry with backoff, and a single token refresh on a 401.
  • Extendable. A new endpoint is a short class (see the last section).

Install

It is a private package, so install it straight from the repo:

npm install git+https://github.com/TravelSommelier/safari-portal-connect.git

Needs Node 18 or newer (it uses the built-in fetch).

Authentication

The API authorizes per user through a browser sign-in (OAuth with PKCE). There is no machine-to-machine login, so you mint tokens once and the SDK keeps them fresh after that.

One-time setup

Run the authorize helper. It opens the Safari Portal sign-in, captures the tokens, and writes them to .tokens.json. 

npm run authorize

Token storage

Tokens live in a TokenStore. The SDK loads them on first use and writes the rotated pair back after every refresh, so the refresh token never goes stale. Use the bundled file store, or implement the interface for Redis, a database, or a secrets manager. 

import { FileTokenStore, type TokenStore } from "safari-portal-connect"

// bundled, file-backed
const tokenStore = new FileTokenStore(".tokens.json")

// or bring your own
class RedisTokenStore implements TokenStore {
  async load() {
    const v = await redis.get("safari:tokens")
    return v ? JSON.parse(v) : null
  }
  async save(tokens) {
    await redis.set("safari:tokens", JSON.stringify(tokens))
  }
}
Keep .tokens.json out of version control. It holds live credentials. It is already in .gitignore.

Creating a client

Pass the token store plus the app credentials. The client id and secret are what let the SDK refresh the access token automatically when it expires. 

import { SafariPortalClient, FileTokenStore } from "safari-portal-connect"

const client = new SafariPortalClient({
  tokenStore: new FileTokenStore(".tokens.json"),
  clientId: process.env.SAFARI_CLIENT_ID,
  clientSecret: process.env.SAFARI_CLIENT_SECRET
})

You can also pass accessToken (and refreshToken) directly instead of a store if you are managing tokens yourself.

Accounts & scoping

Every resource lives under an account. List the accounts your token can act on, then scope to one. Most of the time there is a single account, so firstAccount() is the shortcut. 

// shortcut: scope to the first (usually only) account
const acc = await client.firstAccount()

// or pick one explicitly
const accounts = await client.accounts.list()
const acc = client.account(accounts[0].id)

acc exposes contacts, files, tasks, and users.

Pagination

Same data, three shapes. Pick based on what you need.

Call Returns Use when
list(params) array of every item you want all of it and will map/filter it
stream(params) async generator large sets, or you want to stop early
page(params) one page + meta you need totals or manual paging 
// list: every traveler across all pages, as a plain array
const travelers = await acc.contacts.list({ scope: "travelers" })
const vips = travelers.filter((c) => c.tag_list.includes("vip"))

// stream: lazy, stops fetching the moment you break
for await (const c of acc.contacts.stream({ scope: "travelers" })) {
  if (c.email === target) return c
}

// page: when you want meta (totals, next page)
const page = await acc.contacts.page({ scope: "travelers", limit: 50 })
console.log(page.data, page.meta.total_count, page.hasNextPage)
const next = await page.nextPage()
Heads up: list() fetches every page, so a single call can make several requests. Set limit (max 100) to control page size, and prefer stream() when you only need the first match.

Contacts

Travelers, partners, and suppliers. The type field distinguishes them. Reads return all types; writes are for travelers. 

// read
const jane = await acc.contacts.get(123)
const found = await acc.contacts.search({ q: "jane", scope: "travelers" })
const all = await acc.contacts.list({ scope: "partners" })

// write (travelers)
const created = await acc.contacts.create({
  first_name: "Jane",
  last_name: "Doe",
  email: "jane@example.com",
  tag_list: "vip,honeymoon"
})
await acc.contacts.update(created.id, { preferred_name: "Janie" })

Files

Files are trips or tour requests. They move through a status lifecycle (leads, pipeline, confirmed, completed, archived). New files always start in leads. 

// search within a lifecycle scope
const confirmed = await acc.files.search({ scope: "confirmed" })
const myLeads = await acc.files.search({ scope: "leads", consultant_ids: [11] })

// create and update
const file = await acc.files.create({
  file_name: "Smith Safari 2027",
  priority: "high",
  traveler_ids: [101, 102]
})
await acc.files.update(file.id, { priority: "top", tag_list: "honeymoon,africa" })
traveler_ids replaces, it does not append. The array you send becomes the file's complete traveler set. Omit the key to leave travelers unchanged; send [] to clear them.

Tasks

To-dos attached to files. Read-only: list, search, get.

const overdue = await acc.tasks.search({ scope: "overdue" })
const mine = await acc.tasks.search({ scope: "due-this-week", assigned_to_ids: [11] })
The tasks gotcha. Once you add a filter like file_ids, the endpoint switches to search mode, and search defaults scope to overdue. So "give me this file's tasks" silently returns only the overdue ones. To get them all, sweep the scopes and dedupe: 
const SCOPES = ["overdue", "due-today", "due-this-week", "upcoming", "completed", "tbd"] as const

const byId = new Map()
for (const scope of SCOPES) {
  for (const t of await acc.tasks.search({ scope, file_ids: [fileId] })) {
    byId.set(String(t.id), t)
  }
}
const allTasks = [...byId.values()]

Users

The team members on the account. Read-only. Mostly used to turn the user ids on files and tasks (consultant_id, assigned_to_id) into names. 

const team = await acc.users.list()
const nameById = new Map(team.map((u) => [String(u.id), u.name]))

const alice = await acc.users.get(11)

Categories & stages

Reference lists for turning the id fields on a file into names. Both are read-only and return the whole set in one call (not paginated). categories covers trip types, regions, and lead sources; stages are the pipeline stages a file moves through. 

const categories = await acc.categories.list({ type: "trip_type" })
const stages = await acc.stages.list()

const catName = new Map(categories.map((c) => [String(c.id), c.name]))

const file = await acc.files.get(345151)
const tripType = catName.get(String(file.trip_type_category_id))
These endpoints were renamed. They live at /file_categories and /file_stages; the older /categories and /stages paths now return 404. The accessor names (acc.categories, acc.stages) are unchanged, so the move is invisible from the client.

Invoices & expenses

Two mirror-image resources. Invoices are money coming in (receivable); expenses are money going out to suppliers (payable). Both are read-only: list / page / stream / get, where get returns the full payment breakdown. 

// invoices - needs invoices:read
const invoices = await acc.invoices.list()
const inv = await acc.invoices.get(9)
inv.balance_due // still owed, in minor units
inv.payments // each carries direction "receivable"

// expenses - needs expenses:read
const expenses = await acc.expenses.list()
const exp = await acc.expenses.get(51)
exp.pay_to_id // the supplier contact
exp.payments // each carries direction "payable"
Amounts are in minor units. An amount of 11820400 is 118,204.00 in a two-decimal currency. Divide by 10^(currency decimals) - most are /100, but zero-decimal currencies like JPY and ISK are not. Format with Intl.NumberFormat rather than hardcoding /100.
An expense has no single total or currency. Unlike an invoice, each of an expense's payments carries its own amount and currency, so a supplier payout can span currencies. Aggregate per currency; never sum across.

File events

A file's activity timeline - status changes, proposal views, shares, payments, guest form submissions. Read-only and nested under a file, so it is a method on acc.files rather than a top-level resource. Newest first. 

const events = await acc.files.events(345151, { limit: 100 })

for (const e of events) {
  console.log(e.created_at, e.type, e.user_id)
}

// page() and stream() variants too
const first = await acc.files.eventsPage(345151)
for await (const e of acc.files.eventsStream(345151)) { /* ... */ }
Guest-facing events carry visitor metadata. Events triggered by the end traveler (proposal viewed, form submitted) include request_location and request_ip - the visitor's approximate location and IP, not a team member's. user_id is null for those.

Forms & submissions

Forms are the account's drag-and-drop intake / lead-capture web forms (trip inquiry, guest details, feedback, waivers). Both forms and their submissions are read-only and share one scope, forms:read. The forms API returns metadata only - to see what a form collects, read its submissions. 

const forms = await acc.forms.list()
const form = await acc.forms.get(42)
form.sqid // "wfm_..." public embed key; form.id is the API id

A submission is one completed form. It ties to a contact (and a file, when assigned) and carries a self-contained responses array - each answer includes its own label, type, mapped_to, and value, so you never need the form definition to read it. Reach submissions three ways: 

// by form
const subs = await acc.forms.submissions(42)

// by contact
const theirs = await acc.contacts.formSubmissions(301)

// one submission by id
const one = await acc.formSubmissions.get(9001)

for (const r of one.responses) {
  console.log(r.label, r.value) // r.mapped_to shows the synced contact/file field
}
File answers are signed, expiring links. When a response type is file, its value is { filename, url } where url is a short-lived signed download. Fetch it promptly - it is not stable.
Only completed submissions appear, and a response is a snapshot from submit time, so it can differ from the form's current state if the form was later edited. A submission also outlives a soft-deleted form.

Errors

Every failure is a SafariError subclass. Non-2xx responses throw SafariApiError with the status and a set of helpers.

Class When
SafariApiError non-2xx response. Has status, title, description, and isUnauthorized / isForbidden / isNotFound / isValidationError / isRateLimited / isServerError.
SafariTimeoutError the request passed the configured timeout
SafariNetworkError the underlying fetch failed (DNS, reset, offline)
SafariAuthError a refresh was needed but no tokens were available 
import { SafariApiError } from "safari-portal-connect"

try {
  await acc.contacts.create({ first_name: "", last_name: "" })
} catch (e) {
  if (e instanceof SafariApiError && e.isValidationError) {
    console.log(e.description) // "First name can't be blank, ..."
  } else {
    throw e
  }
}

Config & resilience

Sensible defaults are built in. Override timeout, retry policy, or wire up hooks for logging and token persistence. 

new SafariPortalClient({
  tokenStore,
  clientId,
  clientSecret,
  timeoutMs: 30_000, // per request, default 30s
  retry: { maxRetries: 2, baseMs: 300, maxMs: 5_000 },
  hooks: {
    onRequest: ({ method, url, attempt }) => {},
    onResponse: ({ status, durationMs }) => {},
    onRetry: ({ reason, delayMs }) => log.warn("retrying", reason),
    onTokens: (tokens) => persist(tokens)
  }
})
  • Retries cover network errors, timeouts, and 5xx on safe methods, plus 429 on any method (it respects Retry-After).
  • On a 401 the client refreshes once and retries, then writes the rotated tokens back to the store.

A note on IDs

Connect resource ids are 64-bit, which is bigger than a JavaScript number can hold exactly. The SDK parses them losslessly, so large ids come back as exact strings and smaller ones as numbers. That is why id types are number | string.

You rarely need to think about it. Pass ids straight back into calls and they work. If you key a map or compare ids, normalize with String(id) so a string id and a number id of the same value match. 

const byId = new Map(team.map((u) => [String(u.id), u]))
const user = byId.get(String(task.assigned_to_id))

Adding an endpoint

When the API ships a new resource, adding it is three small steps. The base classes give you list, page, stream, and get for free; WritableResource adds create and update. 

// 1. a model (src/models/note.ts)
export interface Note {
  id: Id
  body: string
}

// 2. a resource (src/resources/notes.ts)
import { ReadableResource } from "../resources/base.js"

export class NotesResource extends ReadableResource<Note> {
  protected readonly resourcePath = "notes"
}

// 3. register it on the account (src/account.ts)
this.notes = new NotesResource(http, accountId)

That is the whole pattern. Use WritableResource instead and set bodyKey if the endpoint supports writes.
